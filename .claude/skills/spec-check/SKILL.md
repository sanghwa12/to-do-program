# spec-check — 기획서 ↔ 코드 정합성 점검

이 프로젝트는 `docs/` 폴더에 기능별 기획서를 두고, 각 기획서의
"요구사항 (검증 항목)" 체크리스트를 실제 코드와 대조해 구현 상태를 점검한다.
**너의 임무는 점검과 보고뿐이다. 코드를 수정하지 마라.**

## 절차

1. **기획서 읽기** — `docs/` 폴더의 모든 `.md`를 읽는다.
   - 단, `README.md`와 `00-template.md`는 제외.
   - 각 문서에서 frontmatter(`기능ID`, `마일스톤`, `상태`)와
     `## 요구사항 (검증 항목)`의 체크리스트 항목(R1, R2, …)을 뽑는다.

2. **코드 읽기** — 프로젝트 소스(`src/`, `public/`, `vite.config.*`, `package.json` 등)를 읽는다.
   - 코드가 아직 없으면(마일스톤 미착수) 그 사실을 그대로 보고한다 — 추측으로 ✅ 하지 마라.

3. **항목별 판정** — 각 요구사항(Rn)마다 다음 중 하나로 판정하고, **반드시 근거(파일:라인)를 단다.**
   - ✅ 구현됨 — 코드에서 해당 구조/동작을 확인함
   - 🚧 부분 — 일부만 구현됨 (무엇이 남았는지 명시)
   - ❌ 미구현 — 코드에 없음
   - ⚠️ 불일치(drift) — 코드가 기획과 **다르게** 구현됨 (무엇이 어떻게 다른지 명시)
   - 근거 없이 ✅ 를 주지 마라. 확인 못 하면 🚧 또는 "확인 불가"로.

4. **역방향 점검** — 코드에는 있으나 **어떤 기획서에도 없는** 주요 기능/화면/파일을 찾아
   "📄 문서화 안 됨"으로 표시한다 (기획서 추가 또는 삭제 검토 대상).

5. **핵심 제약 위반 스캔** — `CLAUDE.md`의 핵심 제약을 코드가 어기는지 확인한다:
   - `localStorage` / `sessionStorage` 사용 여부 (→ IndexedDB만 써야 함)
   - API 키/클라이언트 ID 하드코딩 여부
   - 새 백엔드 서버 / 유료·클라우드 백엔드(Firebase, Supabase 등) 도입 흔적
   위반이 있으면 리포트 최상단에 🚨로 강조한다.

## 출력 형식 (한글, 입문자용)

1. **🚨 위반/불일치 요약** — 제약 위반(🚨)과 drift(⚠️)가 있으면 맨 위에 먼저.
2. **기능별 표** — 문서마다 하나씩:

   ### F0x · (기능명)  — frontmatter 상태: (계획됨 등)
   | 요구사항 | 판정 | 근거 |
   |----------|------|------|
   | R1 …     | ✅   | src/db.js:12 |

3. **전체 요약** — `구현완료 n / 부분 n / 미구현 n / 불일치 n / 문서화안됨 n`.
4. **문서 상태 점검** — frontmatter의 `상태:`와 실제가 어긋난 문서를 지적:
   - (예) "F01 상태=구현완료 인데 R4 미구현 → 상태를 '개발중'으로 낮추길 제안"
   - (예) "F02 모든 항목 ✅ 인데 상태=계획됨 → '구현완료'로 올리길 제안"
   - **문서 파일을 직접 고치지 말고 제안만** 한다 (사용자가 결정).
5. **다음 할 일 제안** — 가장 중요한 미구현/불일치 1~2개를 콕 집어 다음 작업으로 제안.

## 주의
- 규모가 작으면 직접 읽어 점검한다. 코드가 커지면 기능별로 병렬 탐색(Agent)해도 되지만,
  현재 프로젝트는 작으므로 기본은 인라인 점검.
- 판정은 보수적으로. 애매하면 ✅ 대신 🚧/확인불가.
